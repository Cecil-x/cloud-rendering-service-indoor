# 三维控制原型说明

本文记录当前 Three.js 云渲染原型中“无人机控制 / 第一视角小窗 / 遥测显示”的设计和改动点，便于后续迁移到正式客户端页面。

## 1. 当前状态

当前已在测试页面中实现原型：

```text
public/turn.html
public/client-turn.js
public/renderer.js
public/style.css
```

注意：`turn.html` 是当前 TURN 测试页面，不是正式业务客户端页面。后续需要把控制盘、遥测面板、WebSocket 控制消息迁移到正式客户端页面中。

## 2. 功能目标

原型目标：

```text
1. 在 Three.js 服务端渲染场景中显示一个无人机标记
2. 无人机暂时用荧光箭头代替真实模型
3. 支持通过坐标值控制无人机位置
4. 支持控制无人机偏航角
5. 支持控制云台朝向
6. 页面提供控制盘：上、下、左、右、左转、右转、升高、降低
7. 页面实时显示无人机坐标和朝向
8. 视频画面中显示无人机第一视角小窗口
```

## 3. 网络和渲染链路

当前链路仍然保持原云渲染方式：

```text
客户端页面
  -> WebRTC 接收远程视频
  -> WebSocket 发送控制消息

Windows 云渲染主机
  -> Puppeteer 启动 Edge renderer 页面
  -> renderer.js 运行 Three.js 场景
  -> canvas.captureStream() 推送视频
  -> 接收 WebSocket input / drone-control / drone-set
```

视频画面仍然走 WebRTC。无人机控制消息走 WebSocket，不走 WebRTC DataChannel。

## 4. renderer.js 中的服务端渲染改动

### 4.1 无人机状态

服务端 renderer 中维护无人机状态：

```js
const droneState = {
  position: new THREE.Vector3(0, 0.8, 0),
  yaw: 0,
  pitch: 0,
  gimbalYaw: 0,
  gimbalPitch: 0
};
```

字段含义：

```text
position.x / y / z  无人机坐标
yaw                 无人机偏航角，弧度
pitch               无人机俯仰角，弧度，当前预留
gimbalYaw           云台偏航角，弧度
gimbalPitch         云台俯仰角，弧度
```

### 4.2 无人机显示

当前无人机不是模型，而是一个 Three.js 组合体：

```text
ConeGeometry     荧光箭头头部
CylinderGeometry 箭头尾部
TorusGeometry    外圈标识
PointLight       发光效果
```

主要用于在场景中明确标出无人机位置和朝向。

### 4.3 无人机第一视角

服务端新增了一个无人机第一视角相机：

```js
const droneCamera = new THREE.PerspectiveCamera(68, 16 / 9, 0.02, 1000);
```

每次无人机状态变化时，根据无人机位置、偏航角和云台角计算第一视角方向：

```text
lookYaw = droneState.yaw + droneState.gimbalYaw
lookPitch = droneState.pitch + droneState.gimbalPitch
```

### 4.4 小窗渲染

`animate()` 中先渲染主视角，再用 `viewport + scissor` 在右上角渲染无人机第一视角：

```text
主画面：camera
右上角小窗：droneCamera
```

小窗是直接绘制进同一个 canvas 中，因此客户端看到的视频流里会包含这个小窗。

## 5. WebSocket 控制协议

### 5.1 控制盘消息：drone-control

客户端点击控制盘按钮时发送：

```json
{
  "type": "drone-control",
  "action": "forward"
}
```

支持的 `action`：

```text
forward     前进
back        后退
left        左移
right       右移
yaw-left    左转
yaw-right   右转
up          升高
down        降低
```

服务端默认步长：

```text
移动步长：0.25
偏航步长：8 度
```

也可以扩展传入：

```json
{
  "type": "drone-control",
  "action": "forward",
  "step": 0.5,
  "yawStep": 5
}
```

### 5.2 直接设置状态：drone-set

客户端也可以直接设置坐标和朝向：

```json
{
  "type": "drone-set",
  "x": 1.2,
  "y": 0.8,
  "z": -3.4,
  "yaw": 45,
  "gimbalYaw": 0,
  "gimbalPitch": -15
}
```

角度单位：

```text
yaw / pitch / gimbalYaw / gimbalPitch 均使用角度制传入
renderer.js 内部转换为弧度
```

也支持 position 数组：

```json
{
  "type": "drone-set",
  "position": [1.2, 0.8, -3.4],
  "yaw": 45
}
```

### 5.3 遥测消息：telemetry

服务端状态更新后，会向客户端发送：

```json
{
  "type": "telemetry",
  "position": {
    "x": 1.2,
    "y": 0.8,
    "z": -3.4
  },
  "yaw": 45,
  "pitch": 0,
  "gimbalYaw": 0,
  "gimbalPitch": -15
}
```

客户端用它实时更新坐标和朝向显示。

## 8. 本次迭代更新

### 8.1 无人机模型替换

当前原型已从单纯的荧光箭头扩展为优先加载 GLB 模型：

```text
public/models/airplane.glb
```

`renderer.js` 中通过 `GLTFLoader` 加载模型：

```js
import { GLTFLoader } from 'https://unpkg.com/three@0.165.0/examples/jsm/loaders/GLTFLoader.js';
```

加载逻辑：

```text
1. 先创建荧光箭头作为 fallback
2. 异步加载 /models/airplane.glb
3. GLB 加载成功后替换箭头
4. GLB 加载失败则继续使用箭头
```

模型加载后会自动做：

```text
1. 通过 Box3 计算包围盒
2. 根据包围盒中心居中模型
3. 根据最大尺寸统一缩放
4. 修正模型朝向
5. 挂载到无人机根节点下
```

当前模型挂载结构：

```text
drone              # 无人机根节点，负责坐标和偏航旋转
  └─ modelPivot    # 模型修正节点，负责缩放和模型初始朝向修正
       └─ model    # GLB 模型本体，已按包围盒中心居中
```

这样做的目的是避免模型自身 pivot 不在几何中心时，左转/右转出现绕外部圆心旋转的问题。

### 8.2 模型旋转中心修正

之前模型直接挂到 `drone` 下，同时对模型做 `position / scale / rotation` 修正，可能导致模型视觉中心和旋转中心不一致。

现在改为：

```js
model.position.sub(center);

const modelPivot = new THREE.Group();
modelPivot.scale.setScalar(0.9 / maxSize);
modelPivot.rotation.y = Math.PI;
modelPivot.add(model);

drone.clear();
drone.add(modelPivot);
```

其中：

```text
drone.position      表示无人机真实坐标
drone.rotation.y    表示无人机偏航角
modelPivot          只负责模型自身尺寸和朝向修正
model.position      只负责把模型几何中心移到局部原点
```

### 8.3 左右移动方向修正

发现问题：无人机左转 90 度后，左移和右移方向反了。

原因：偏航角的正方向和移动向量计算方向不一致。

修正后的方向向量：

```js
function updateDroneVectors() {
  droneDirection.set(-Math.sin(droneState.yaw), 0, -Math.cos(droneState.yaw)).normalize();
  droneRight.set(Math.cos(droneState.yaw), 0, -Math.sin(droneState.yaw)).normalize();
}
```

同时同步修正无人机第一视角方向：

```js
const lookDirection = new THREE.Vector3(
  -Math.sin(lookYaw) * cp,
  Math.sin(lookPitch),
  -Math.cos(lookYaw) * cp
).normalize();
```

修正后的语义：

```text
yaw = 0          朝向 -Z
左转 90 度       朝向 -X
forward          沿当前机头方向移动
left/right       沿当前机体左右方向平移
第一视角小窗      与机头/云台方向保持一致
```

### 8.4 自动旋转关闭

已关闭点云/模型的自动旋转逻辑，避免无人机控制测试时场景持续自转影响判断。

当前 `animate()` 中只负责渲染主视角和无人机第一视角小窗，不再执行自动旋转。

## 10. 最近更新：标定与实时数据接入

### 10.1 点云标定方式调整

```json

  "x": 0.5,
  "y": 0.5
}
```

```json
{
  "type": "calibration-use-current-heading"
}
```

### 10.2 实时无人机数据接入

测试页面新增实时状态 WebSocket 输入框，当前可连接：

```text
ws://114.116.235.66/indoorUavFlightControlBackend/siteWebSocketServer/4a77dec9
```

客户端收到后端实时数据后，会通过原有 `/signal` 通道转发给服务端 renderer：

```json
{
  "type": "uav-realtime",
  "payload": {
    "data": {
      "horizontalX": "0",

```

服务端 renderer 根据标定参数将无人机实时坐标转换到点云场景坐标。

### 10.3 当前坐标映射

现场实测结论：

```text
无人机起飞点 = 实时坐标 (0, 0, 0)
起飞朝向 = 无人机 X 正方向
无人机 Y 正方向 = 机体左侧
```

当前标定完成后的映射逻辑：

```text
scenePosition =
  mapOrigin
  + xAxis * horizontalX
  - rightAxis * horizontalY
  + upAxis * horizontalZ
```

含义：

```text
horizontalX 增大：沿标定后的 X 正方向前进
horizontalY 增大：向机体左侧移动
horizontalZ 增大：向上移动
```

偏航角映射：

```text
sceneYaw = 标定 X 方向角 + uavYaw
```

实际飞行验证中发现 `uavYaw` 与画面偏航存在偏差，当前已调整为优先使用：

```text
uavYawForBoard
```

如果后端数据中没有 `uavYawForBoard`，再回退使用：

```text
uavYaw
```

云台俯仰：

```text
gimbalPitch = 后端 gimbalPitch
```

### 10.4 非 JSON 欢迎消息处理

站点 WebSocket 连接后会先返回普通文本欢迎语，例如：

```text
站点WebSocket连接成功，欢迎【站点密钥】加入连接
```

客户端已增加判断：只有消息以 `{` 或 `[` 开头时才按 JSON 解析；普通文本只显示为连接提示，不再抛出 `JSON.parse` 异常。

### 10.5 点云显示比例与均匀抽样优化

为缓解接收实时数据后画面卡顿的问题，测试页面增加了“点云显示比例”滑块：

```text
范围：5% - 100%
默认：20%
步进：5%
```

该滑块通过 WebSocket 发送：

```json
{
  "type": "point-density",
  "percent": 20
}
```

服务端 renderer 接收后调用：

```js
geometry.setDrawRange(0, visibleCount);
```

因此调节显示比例时：

```text
不会重新加载点云文件
不会重新解析 PLY/PCD
会立即影响主画面和无人机第一视角小窗
```

当前最大显示点数按原始点云点数计算：

```text
100% = 原始点云全部点
20%  = 原始点云 20%
5%   = 原始点云 5%
```

例如 `map_aligned_icp_baked.ply` 约 1500 万点：

```text
100% ≈ 1500 万点
20%  ≈ 300 万点
5%   ≈ 75 万点
```

为避免简单 `drawRange(0, N)` 只显示原始点数组前 N 个点导致分布不均，renderer 已增加均匀绘制顺序：

```js
applyUniformPointOrder(geometry);
```

实现方式：

```text
1. 点云加载后生成一份 index buffer
2. 使用与点数互质的 stride 遍历原始点数组
3. 将点云绘制顺序打散成较均匀的空间抽样顺序
4. 滑块调节时仍然只改 drawRange，不重建 geometry
```

这样 `20% / 50% / 100%` 调节时，显示点会尽量分布在整个点云范围内，而不是只截取文件开头的一部分。

之前曾使用固定点数上限：

```js
const POINT_BUDGET = 2_000_000;
```

当前已调整为：

```js
const POINT_BUDGET = Infinity;
```

即默认不再加载时强制降到 200 万点，而是由页面滑块控制实际显示比例。

如仍然卡顿，建议优先将页面滑块降到：

```text
5% - 20%
```

如果需要同时降低首次加载耗时和内存占用，才考虑重新启用固定 `POINT_BUDGET`，例如：

```js
const POINT_BUDGET = 1_000_000;
// 或
const POINT_BUDGET = 500_000;
```


最后一个客户端断开后，服务端会在超时后关闭 renderer WebSocket 和 Puppeteer page。

### 10.7 渲染帧率显示

为便于观察实时渲染性能，服务端 renderer 已在视频画面左上角增加 FPS 显示。

实现方式：

```text
1. renderer.js 中新增 overlayScene 和 OrthographicCamera
2. 使用 canvas 绘制 FPS 文本
3. 将 canvas 转为 CanvasTexture
4. 用 Sprite 叠加到最终渲染画面左上角
5. 约每 500ms 更新一次 FPS 数值
```

由于同一帧中需要依次渲染主画面、无人机第一视角小窗和 FPS 叠加层，已关闭 Three.js 自动清屏：

```js
renderer.autoClear = false;
```

并在每帧开始时手动清屏：

```js
renderer.clear();
```

当前渲染顺序：

```text
1. 手动清屏
2. 渲染主视角
3. 渲染无人机第一视角小窗
4. 渲染 FPS overlay
```


```text

```

这样重启服务或重新进入页面后，不需要每次重新点击起飞点和设置 X 正方向。

### 10.9 无人机飞行动画

当前无人机 GLB 模型已接入模型自带动画。`public/models/airplane.glb` 中包含的动画会在模型加载成功后创建 `AnimationMixer`。

动画播放策略：

```text
1. 无人机位置或姿态发生变化时播放动画
2. 无人机停止移动后一段时间自动暂停动画
3. GLB 加载失败时仍回退使用荧光箭头，不影响基础控制
```

当前主要用于表现螺旋桨旋转，使实时飞行和手动控制时的无人机状态更直观。

### 10.10 当前任务航线绘制

服务端已增加飞控平台接口代理，用于在 renderer 中绘制任务航线。客户端仍只访问本地服务端接口，飞控平台登录和 token 处理保留在 `server.js` 中，避免把账号和 token 暴露到前端。

本地接口：

```text
GET /api/uav/running-route
GET /api/uav/running-route?missionId=xxx
GET /api/uav/running-route?flightStrategyId=xxx
GET /api/uav/available-routes
```

服务端当前流程：

```text
1. 调用飞控平台 thirdPartyLogin 获取 token
2. 查询任务列表
3. /api/uav/available-routes 过滤所有 missionState = 1 且存在 flightStrategyId / strategyId / flightHorizontalStrategyId 的任务
4. 前端从下拉框选择具体任务航线
5. /api/uav/running-route 根据 missionId 或 flightStrategyId 定位任务
6. 根据 flightStrategyId 查询航线详情
7. 返回水平航点和垂直航点给 renderer
```

`/api/uav/available-routes` 返回结构：

```json
{
  "ok": true,
  "routes": [
    {
      "id": 1,
      "value": "1",
      "flightStrategyId": 100,
      "missionName": "任务名称",
      "missionState": 1,
      "historyState": null,
      "historyStateDescription": null,
      "state": null,
      "status": null
    }
  ]
}
```

renderer 默认会定时轮询 `/api/uav/running-route`。正式前端也可以通过 `/signal` 发送航线切换消息，让 renderer 改为轮询指定任务航线：

```json
{
  "type": "route-select",
  "url": "/api/uav/running-route?missionId=xxx"
}
```

清除航线时发送空地址：

```json
{
  "type": "route-select",
  "url": ""
}
```

renderer 会将航线点按标定坐标系转换到点云场景中绘制：

```text
1. 绘制水平航线
2. 绘制垂直航线
3. 绘制航点标记
4. 绘制航点编号/标签
```

注意：已发现 `missionState = 1` 表示“已启用任务”，不是“正在执行任务”。因此当前正式前端不再把第一个 `missionState = 1` 任务直接当作正在飞行任务，而是先通过 `/api/uav/available-routes` 展示所有可用航线，由用户在下拉框中选择需要绘制的航线。后续如果飞控平台提供明确的执行中状态字段，可以再恢复自动选择当前执行任务。

## 11. 注意事项

- 当前小窗第一视角是画在服务端 canvas 里的，不是客户端额外渲染。
- 当前控制消息通过 WebSocket `/signal` 复用原信令通道。
- 当前原型优先改在 TURN 测试页面，正式页面需要单独迁移 UI 和 JS。
- 当前无人机 GLB 模型用于原型验证，模型尺寸和朝向通过代码做了自动居中、缩放和朝向修正。
- 如果更换 GLB 模型，重点检查 `modelPivot.scale` 和 `modelPivot.rotation.y`。
- 如果模型仍有轻微偏心，说明模型几何包围盒中心和真实重心不一致，可在 `model.position` 上增加人工偏移。
- 如果使用远程客户端跨网段访问，仍建议使用 `/turn.html` 和 TURN 模式。
- 当前实时坐标映射假设 `horizontalY` 为机体左侧正方向，如果现场数据定义变化，需要调整 `renderer.js` 中的 `horizontalY` 符号。
- 当前实时数据通过测试页面连接后端 WebSocket，正式客户端接入时建议封装成独立数据服务或统一状态管理模块。
- 当前航线绘制依赖标定坐标系，建议先确认 `origin.txt` 已正确加载，再判断航线位置是否偏移。
- 当前 `missionState = 1` 只能视为任务已启用，不能视为正在执行；正式前端通过下拉框选择具体航线，运行中任务自动筛选逻辑还需要结合飞控平台真实状态字段继续修正。
